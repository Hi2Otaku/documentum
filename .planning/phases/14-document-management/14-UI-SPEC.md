## Interaction Contracts

### File Upload (DOC-01, D-03, D-04)

1. User drags files onto the drop zone OR clicks the "Browse" button to open file picker.
2. Multiple files accepted. Each file creates a separate document via `POST /api/documents/`.
3. API calls execute sequentially (not parallel) to avoid overwhelming the server.
4. During upload, drop zone transforms to show `UploadProgressItem` for each file.
5. Each item shows: filename, Progress bar (indeterminate until response), then success/error icon.
6. On success: invalidate `["documents"]` query, show toast "Document uploaded" per file.
7. On error: show error icon on that file row + toast with error message. Other files continue uploading.
8. After all uploads complete, progress list auto-dismisses after 3 seconds, drop zone returns to default state.
9. Title defaults to filename (without extension). Author defaults to current user's username.

### Table Row Selection

1. User clicks a table row -> row becomes selected (highlighted background + left border).
2. Detail panel loads that document's details via `GET /api/documents/{id}`.
3. Only one row selected at a time.
4. Selected row state stored in component state (not URL, not Zustand).
5. On page change (pagination), selection clears and detail panel shows "Select a document".

### Filter Bar (DOC-02)

1. Three filter controls above table:
   - Title: Input with placeholder "Search by title...". Debounced 300ms.
   - Author: Input with placeholder "Filter by author...". Debounced 300ms.
   - State: Select dropdown with options: "All", "Draft", "Review", "Approved", "Archived".
2. Default: all filters empty / "All" state.
3. Changing any filter resets pagination to page 1 and clears selection.
4. Filter values passed as query params to `GET /api/documents?title=&author=&page=&page_size=`.
5. State filter is client-side (API does not have a lifecycle_state filter param -- filter the response data).

### Check Out (DOC-04, D-06)

1. Detail panel shows "Check Out" button when document is NOT locked.
2. Click button -> immediate `POST /api/documents/{id}/checkout` (no dialog needed -- low-risk, reversible).
3. On success: optimistic update -- LockIndicator shows "Checked out by you", Check Out button hides, Check In + Cancel Checkout buttons appear. Toast: "Document checked out".
4. On error: toast with error message.

### Check In (DOC-04, D-07)

1. Detail panel shows "Check In" button when document is locked by the current user.
2. Click opens CheckInDialog.
3. Dialog contents: "Check In Document" heading, file picker button ("Select new version"), selected filename display, optional comment Textarea (placeholder: "Version comment (optional)"), Cancel + "Check In" buttons.
4. "Check In" button disabled until a file is selected.
5. Calls `POST /api/documents/{id}/checkin` with multipart form data (file + comment).
6. On success: close dialog, invalidate document + versions queries, show toast "New version checked in". Lock releases automatically.
7. On error: keep dialog open, show error in toast.

### Cancel Checkout

1. Detail panel shows "Cancel Checkout" button when document is locked by the current user.
2. Click opens a small confirmation dialog: "Cancel checkout? Your lock will be released and no new version will be created."
3. Confirm calls `POST /api/documents/{id}/unlock` (same endpoint as admin force-unlock, but user unlocks their own).
4. On success: lock indicator clears, buttons reset. Toast: "Checkout cancelled".
5. On error: toast with error message.

### Lifecycle Transition (DOC-05, D-08, D-09)

1. Detail panel shows current lifecycle state as a LifecycleStateBadge.
2. Next to the badge, a Select dropdown lists valid transitions:
   - Draft -> Review
   - Review -> Approved
   - Review -> Draft (reject back)
   - Approved -> Archived
3. Selecting a target state opens LifecycleTransitionDialog.
4. Dialog contents: "Change Lifecycle State" heading, "Current: [badge]" -> "New: [badge]" visual, Cancel + "Confirm Transition" buttons.
5. Confirm calls `POST /api/documents/{id}/lifecycle/transition` with `{ target_state: selectedState }`.
6. On success: close dialog, update document query, show toast "State changed to [new state]".
7. On error: keep dialog open, show error in toast.

### Version History (DOC-03, D-05)

1. Version history section in detail panel shows all versions via `GET /api/documents/{id}/versions`.
2. List sorted newest first (by created_at descending).
3. Each version row shows: version label (e.g. "v1.2", 14px semibold), created date (12px muted), author (12px muted), content hash truncated to 8 chars (12px mono muted), download button (Lucide `Download` icon button).
4. Download button triggers `GET /api/documents/{id}/versions/{version_id}/download`. Browser handles the file download via Content-Disposition header.
5. Loading state: 3 Skeleton rows.

### Pagination

Inherited from Phase 13 pattern.

1. Page size: 20 items (matches API default).
2. Pagination controls at table footer: "Page X of Y", Previous/Next buttons.
3. Previous disabled on page 1. Next disabled on last page.
4. Page change triggers `GET /api/documents?page={page}&page_size=20&title={filter}&author={filter}`.

### Loading States

1. Table loading: 5 Skeleton rows, each 48px tall, spanning all columns.
2. Detail panel loading: Skeleton blocks matching header (1 line), metadata card (6 lines), actions (button-sized blocks), version list (3 rows).
3. Initial page load: both table and detail panel show skeletons simultaneously.

## State Management

### Server State (TanStack Query)

| Query Key | Endpoint | Refetch |
|-----------|----------|---------|
| `["documents", { page, title, author }]` | `GET /api/documents` | On filter change, on page change, after upload/mutation |
| `["documents", documentId]` | `GET /api/documents/{id}` | On row selection, after checkout/checkin/lifecycle mutation |
| `["documents", documentId, "versions"]` | `GET /api/documents/{id}/versions` | On document selection, after checkin mutation |

### Mutations

| Mutation | Endpoint | Invalidates |
|----------|----------|-------------|
| uploadDocument | `POST /api/documents/` | `["documents"]` |
| checkoutDocument | `POST /api/documents/{id}/checkout` | `["documents"]`, `["documents", id]` |
| checkinDocument | `POST /api/documents/{id}/checkin` | `["documents"]`, `["documents", id]`, `["documents", id, "versions"]` |
| cancelCheckout | `POST /api/documents/{id}/unlock` | `["documents"]`, `["documents", id]` |
| transitionLifecycle | `POST /api/documents/{id}/lifecycle/transition` | `["documents"]`, `["documents", id]` |

### Client State (Component-level)

| State | Type | Scope |
|-------|------|-------|
| selectedDocumentId | `string | null` | DocumentsPage component |
| titleFilter | `string` | DocumentsPage component |
| authorFilter | `string` | DocumentsPage component |
| stateFilter | `string` | DocumentsPage component |
| currentPage | `number` | DocumentsPage component |
| uploadingFiles | `Array<{ name: string, progress: number, status: 'pending' | 'uploading' | 'done' | 'error' }>` | DocumentsPage component |
| isDragOver | `boolean` | DocumentDropZone component |

No new Zustand stores needed. All document page state is local to the component tree.
