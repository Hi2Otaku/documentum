## Interaction Contracts

### Start Workflow Wizard (WF-01, D-01 through D-04)

1. User clicks "Start Workflow" button in toolbar.
2. StartWorkflowDialog opens at Step 1 (Template selection).
3. Step 1: Fetch installed templates via `GET /api/v1/templates/` (filter client-side for `state === "installed"`). User selects one template card. "Next" button disabled until a template is selected.
4. Step 2: Fetch documents via `GET /api/documents/`. Display with checkboxes. User can select zero or more documents. "Next" always enabled (documents optional).
5. Step 3: Fetch template detail via `GET /api/v1/templates/{id}` to get process variables. Render type-appropriate inputs pre-filled with default values. "Next" always enabled (variables have defaults).
6. Step 4: Review summary. "Launch" button triggers `POST /api/workflows` with `{ template_id, document_ids, initial_variables }`.
7. "Back" button navigates to previous step (disabled on Step 1).
8. On launch success: close dialog, invalidate `["workflows"]` query, show toast "Workflow started". Select the new instance in the table.
9. On launch error: keep dialog open, show error in toast.
10. Closing dialog (X button or escape) resets wizard state.

### Instance Table & Selection (WF-02, D-05 through D-07)

1. On page load: fetch workflow instances.
   - Regular users: `GET /api/workflows?skip=0&limit=20` (returns only their instances per API behavior).
   - Admins: `GET /api/workflows/admin/list?skip=0&limit=20` (returns all instances with enriched metadata).
2. Admin endpoint supports query params: `state`, `template_id`, `date_from`, `date_to`. Regular endpoint supports `skip` and `limit` only.
3. User clicks a table row -> row becomes selected (highlighted background + left border).
4. Detail panel loads that workflow's details via `GET /api/workflows/{id}`.
5. Only one row selected at a time.
6. Selected row state stored in component state (not URL, not Zustand).
7. On page change (pagination), selection clears and detail panel shows "Select a workflow".

### Filter Bar (WF-02, D-07)

1. Two filter controls in toolbar (right side):
   - Template: Select dropdown listing installed templates. Default "All templates".
   - State: Select dropdown with options: "All states", "Running", "Halted", "Finished", "Failed", "Dormant".
2. For admins: filters are sent as query params to admin list endpoint.
3. For regular users: filters are applied client-side (regular list endpoint lacks filter params).
4. Changing any filter resets pagination to page 1 and clears selection.

### Admin Actions (WF-03, D-11 through D-13)

#### Halt (D-13)
1. Detail panel shows "Halt" button when workflow state is RUNNING and user is admin.
2. Click button -> immediate `POST /api/workflows/{id}/halt` (no dialog -- reversible action).
3. On success: invalidate workflow queries, show toast "Workflow halted".
4. On error: toast with error message.

#### Resume (D-13)
1. Detail panel shows "Resume" button when workflow state is HALTED and user is admin.
2. Click button -> immediate `POST /api/workflows/{id}/resume` (no dialog -- reversible action).
3. On success: invalidate workflow queries, show toast "Workflow resumed".
4. On error: toast with error message.

#### Terminate (D-12)
1. Detail panel shows "Terminate" button when workflow state is RUNNING or HALTED and user is admin.
2. Click opens TerminateDialog.
3. Dialog contents: "Terminate Workflow" heading, warning text "This cannot be undone. All pending work items will be cancelled.", text input with placeholder "Type TERMINATE to confirm", "Cancel" + "Terminate" buttons.
4. "Terminate" button disabled until input value === "TERMINATE" (case-sensitive).
5. "Terminate" button uses destructive variant (red fill).
6. Calls `POST /api/workflows/{id}/abort`.
7. On success: close dialog, invalidate workflow queries, show toast "Workflow terminated".
8. On error: keep dialog open, show error in toast.

### Progress Graph (WF-04, D-08 through D-10)

1. When user selects the "Progress" tab in detail panel, render WorkflowProgressGraph.
2. Fetch workflow detail via `GET /api/workflows/{id}` which includes `activity_instances` with state.
3. Fetch template detail via `GET /api/v1/templates/{template_id}` to get activity templates (positions, types) and flow templates (edges).
4. Map template activities to React Flow nodes. Map template flows to React Flow edges.
5. Apply dagre auto-layout using existing `getLayoutedElements` function (LR direction).
6. For each node, look up the corresponding activity instance and apply state-colored border from the activity state color table.
7. Nodes use the existing node components (StartNode, ManualNode, AutoNode, EndNode) but wrapped to override border colors based on activity state. Handles are hidden (no source/target circles -- read-only mode).
8. React Flow configured with: `nodesDraggable={false}`, `nodesConnectable={false}`, `elementsSelectable={false}`, `panOnDrag={true}`, `zoomOnScroll={true}`, `fitView={true}`.
9. MiniMap and Controls are NOT shown (graph is small enough at 300px height).

### Pagination

Inherited from Phase 13/14 pattern.

1. Page size: 20 items (matches API default).
2. Pagination controls at table footer: "Page X of Y", Previous/Next buttons.
3. Previous disabled on page 1. Next disabled on last page.
4. Page change triggers API call with updated `skip` param.

### Loading States

1. Table loading: 5 Skeleton rows, each 48px tall, spanning all columns.
2. Detail panel loading: Skeleton blocks matching header (1 line), metadata card (6 lines), tabs (2 skeleton blocks).
3. Wizard loading (template list in Step 1): 3 Skeleton cards.
4. Progress graph loading: Single Skeleton block, 300px height.
5. Initial page load: both table and detail panel show skeletons simultaneously.

## State Management

### Server State (TanStack Query)

| Query Key | Endpoint | Refetch |
|-----------|----------|---------|
| `["workflows", { page, state, templateId }]` | `GET /api/workflows` (regular) or `GET /api/workflows/admin/list` (admin) | On filter change, on page change, after start/halt/resume/terminate |
| `["workflows", workflowId]` | `GET /api/workflows/{id}` | On row selection, after admin action mutation |
| `["templates"]` | `GET /api/v1/templates/` | On wizard open (Step 1) |
| `["templates", templateId]` | `GET /api/v1/templates/{id}` | On wizard Step 3 (for variables), on Progress tab (for graph structure) |
| `["documents"]` | `GET /api/documents/` | On wizard Step 2 |

### Mutations

| Mutation | Endpoint | Invalidates |
|----------|----------|-------------|
| startWorkflow | `POST /api/workflows` | `["workflows"]` |
| haltWorkflow | `POST /api/workflows/{id}/halt` | `["workflows"]`, `["workflows", id]` |
| resumeWorkflow | `POST /api/workflows/{id}/resume` | `["workflows"]`, `["workflows", id]` |
| terminateWorkflow | `POST /api/workflows/{id}/abort` | `["workflows"]`, `["workflows", id]` |

### Client State (Component-level)

| State | Type | Scope |
|-------|------|-------|
| selectedWorkflowId | `string | null` | WorkflowsPage component |
| templateFilter | `string` ("all" or template ID) | WorkflowsPage component |
| stateFilter | `string` ("all" or state value) | WorkflowsPage component |
| currentPage | `number` | WorkflowsPage component |
| wizardOpen | `boolean` | WorkflowsPage component |
| wizardStep | `number` (1-4) | StartWorkflowDialog component |
| selectedTemplateId | `string | null` | StartWorkflowDialog component |
| selectedDocumentIds | `string[]` | StartWorkflowDialog component |
| variableValues | `Record<string, unknown>` | StartWorkflowDialog component |
| terminateDialogOpen | `boolean` | WorkflowDetailPanel component |
| terminateInput | `string` | TerminateDialog component |
| activeTab | `string` ("details" or "progress") | WorkflowDetailPanel component |

No new Zustand stores needed. All workflow page state is local to the component tree.
