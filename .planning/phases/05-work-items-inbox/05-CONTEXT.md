# Phase 5: Work Items & Inbox - Context

**Gathered:** 2026-03-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Users have a functional inbox where they can view, complete, and reject tasks generated by the workflow engine, with basic performer assignment (supervisor, specific user, group) routing tasks to the right users. Covers INBOX-01 through INBOX-07 and PERF-01 through PERF-03.

NOT in scope: Sequential performers (PERF-04), runtime selection (PERF-05), alias sets (ALIAS-01/02/03), delegation (INBOX-08), work queues (QUEUE-01-04). Those are Phases 6 and 10.

</domain>

<decisions>
## Implementation Decisions

### Inbox API Design
- **D-01:** Both dedicated /inbox and generic /work-items endpoints — dedicated GET /api/v1/inbox for user-facing inbox view with rich context, generic work-items endpoints for admin queries.
- **D-02:** Nested response — each inbox item includes: work item fields + activity template info (name, type, instructions) + workflow summary (template name, state) + document list (id, title, version). One API call gets everything.
- **D-03:** Essential filters — filter by: state (pending/completed), priority, workflow template name. Sort by: priority, due_date, created_at. Covers INBOX-02 and INBOX-07.

### Rejection & Comments
- **D-04:** Mark + defer reject flows — POST /inbox/{id}/reject marks work item as REJECTED (add new state to WorkItemState enum), records in audit trail, but does NOT trigger reject flow routing. Phase 6 adds flow-based routing.
- **D-05:** Dedicated work_item_comments table — new table with id, work_item_id, user_id, content, created_at. Supports multiple comments per work item with proper foreign keys.

### Performer Assignment
- **D-06:** Group-based = one work item per member — when a group is the performer (PERF-03), create a separate work item for each group member. All must complete (parallel). Matches Documentum's group performer behavior.
- **D-07:** Performer resolution in engine service — add resolve_performers() to engine_service.py that maps performer_type + performer_id to actual user IDs. Called during activity activation. Three rules: SUPERVISOR -> workflow initiator, USER -> specific user, GROUP -> all group members.

### Work Item Lifecycle
- **D-08:** Require explicit claiming — POST /inbox/{id}/acquire locks item to user (AVAILABLE -> ACQUIRED). Must acquire before completing or rejecting. Prevents concurrent work on same item.
- **D-09:** Allow release — POST /inbox/{id}/release returns ACQUIRED -> AVAILABLE, clearing performer_id. User can unclaim if they changed their mind.

### Claude's Discretion
- WorkItemComment model schema design details
- Inbox Pydantic schema structure (nesting depth, field selection)
- How to handle the existing complete_work_item in engine_service (extend vs wrap)
- Migration strategy for new REJECTED state and comments table
- Test fixture design for inbox scenarios

</decisions>

## Existing Code Insights

### Reusable Assets
- `src/app/models/workflow.py` — WorkItem model already exists with performer_id, state, priority, due_date, instructions, completed_at
- `src/app/models/enums.py` — WorkItemState (ACQUIRED, AVAILABLE, DELEGATED, COMPLETE), PerformerType (USER, GROUP, SUPERVISOR, ALIAS)
- `src/app/services/engine_service.py` — complete_work_item already advances workflow, creates work items during advancement
- `src/app/routers/workflows.py` — Has GET /workflows/{id}/work-items and POST /work-items/{id}/complete
- `src/app/models/base.py` — BaseModel with UUID, timestamps, soft delete
- `src/app/services/audit_service.py` — Audit trail integration
- `src/app/schemas/common.py` — EnvelopeResponse, PaginatedResponse patterns

### Established Patterns
- Service layer: routers delegate to service functions
- Async SQLAlchemy with selectinload for eager loading
- UUID PKs, soft deletes, audit on every mutation
- EnvelopeResponse wrapping, offset pagination
- Pydantic schemas for all request/response

### Integration Points
- Inbox router registers at /api/v1/inbox in main.py
- Inbox service calls engine_service.complete_work_item for completion (not reimplemented)
- WorkItemComment model needs new table + Alembic migration
- REJECTED state addition to WorkItemState enum needs migration
- Performer resolution hooks into engine_service._advance_from_activity where work items are created

### Key Model Notes
- WorkItem already has viewonly relationship to WorkflowInstance via ActivityInstance
- WorkItem.performer_id is nullable (supports unassigned items)
- ActivityTemplate has performer_type and performer_id fields for configuration
- User model has group memberships via user_groups junction table
