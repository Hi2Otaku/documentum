# Phase 5: Work Items & Inbox - Research

**Researched:** 2026-03-30
**Domain:** Inbox API, performer resolution, work item lifecycle, comments
**Confidence:** HIGH

## Summary

Phase 5 adds a user-facing inbox for work items generated by the Phase 4 engine, plus performer resolution logic (supervisor, user, group) and work item comments. The existing codebase already has a WorkItem model, WorkItemState enum, and a complete_work_item function in engine_service.py. The main work is: (1) new inbox router with rich nested responses, (2) inbox service with filtering/sorting/claiming, (3) performer resolution in the engine advancement loop, (4) WorkItemComment model and migration, (5) REJECTED state addition.

This is a backend-only API phase. All code follows established patterns: async SQLAlchemy with selectinload, service layer, Pydantic schemas, EnvelopeResponse wrapping, audit on mutations.

**Primary recommendation:** Extend the existing engine_service with a resolve_performers() function and create a dedicated inbox_service.py + inbox router. Do not duplicate complete_work_item logic -- the inbox service should delegate to engine_service for completion and advancement.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Both dedicated /inbox and generic /work-items endpoints -- dedicated GET /api/v1/inbox for user-facing inbox view with rich context, generic work-items endpoints for admin queries.
- **D-02:** Nested response -- each inbox item includes: work item fields + activity template info (name, type, instructions) + workflow summary (template name, state) + document list (id, title, version). One API call gets everything.
- **D-03:** Essential filters -- filter by: state (pending/completed), priority, workflow template name. Sort by: priority, due_date, created_at. Covers INBOX-02 and INBOX-07.
- **D-04:** Mark + defer reject flows -- POST /inbox/{id}/reject marks work item as REJECTED (add new state to WorkItemState enum), records in audit trail, but does NOT trigger reject flow routing. Phase 6 adds flow-based routing.
- **D-05:** Dedicated work_item_comments table -- new table with id, work_item_id, user_id, content, created_at. Supports multiple comments per work item with proper foreign keys.
- **D-06:** Group-based = one work item per member -- when a group is the performer (PERF-03), create a separate work item for each group member. All must complete (parallel). Matches Documentum's group performer behavior.
- **D-07:** Performer resolution in engine service -- add resolve_performers() to engine_service.py that maps performer_type + performer_id to actual user IDs. Called during activity activation. Three rules: SUPERVISOR -> workflow initiator, USER -> specific user, GROUP -> all group members.
- **D-08:** Require explicit claiming -- POST /inbox/{id}/acquire locks item to user (AVAILABLE -> ACQUIRED). Must acquire before completing or rejecting. Prevents concurrent work on same item.
- **D-09:** Allow release -- POST /inbox/{id}/release returns ACQUIRED -> AVAILABLE, clearing performer_id. User can unclaim if they changed their mind.

### Pattern 1: Service Delegation (not duplication)
**What:** inbox_service calls engine_service.complete_work_item for completion logic rather than reimplementing advancement.
**When to use:** Any time inbox actions affect workflow state.
**Example:**
```python
# src/app/services/inbox_service.py
async def complete_inbox_item(
    db: AsyncSession,
    work_item_id: uuid.UUID,
    user_id: str,
    output_variables: dict[str, Any] | None = None,
) -> WorkItem:
    """Complete a work item from the inbox. Delegates to engine_service."""
    # 1. Load work item to get workflow_instance_id
    work_item = await _load_work_item(db, work_item_id)
    # 2. Verify user owns the acquired item
    if work_item.state != WorkItemState.ACQUIRED:
        raise ValueError("Work item must be acquired before completing")
    if str(work_item.performer_id) != user_id:
        raise ValueError("Only the assigned performer can complete this item")
    # 3. Get workflow_id via activity_instance
    activity = work_item.activity_instance
    # 4. Delegate to engine_service
    return await engine_service.complete_work_item(
        db, activity.workflow_instance_id, work_item_id, user_id, output_variables
    )
```

### Pattern 2: Rich Nested Response via selectinload Chain
**What:** Single query loads work item + activity instance + activity template + workflow instance + process template + workflow packages, all via SQLAlchemy eager loading.
**When to use:** GET /inbox endpoint to satisfy D-02 (one API call gets everything).
**Example:**
```python
query = (
    select(WorkItem)
    .options(
        selectinload(WorkItem.activity_instance)
        .selectinload(ActivityInstance.activity_template),
        selectinload(WorkItem.activity_instance)
        .selectinload(ActivityInstance.workflow_instance)
        .selectinload(WorkflowInstance.process_template),
        selectinload(WorkItem.activity_instance)
        .selectinload(ActivityInstance.workflow_instance)
        .selectinload(WorkflowInstance.workflow_packages),
    )
    .where(
        WorkItem.performer_id == user_uuid,
        WorkItem.is_deleted == False,
    )
)
```

### Pattern 3: Performer Resolution in Engine
**What:** resolve_performers() maps (performer_type, performer_id, workflow) to list of user UUIDs. Called during _advance_from_activity when creating work items for manual activities.
**When to use:** Every time the engine activates a manual activity.
**Example:**
```python
async def resolve_performers(
    db: AsyncSession,
    performer_type: str | None,
    performer_id: str | None,
    workflow: WorkflowInstance,
) -> list[uuid.UUID]:
    """Resolve performer config to actual user IDs."""
    if not performer_type:
        return []

    match performer_type.lower():
        case "supervisor":
            if workflow.supervisor_id:
                return [workflow.supervisor_id]
            return []
        case "user":
            if performer_id:
                return [uuid.UUID(performer_id)]
            return []
        case "group":
            if not performer_id:
                return []
            # Query group members
            result = await db.execute(
                select(user_groups.c.user_id).where(
                    user_groups.c.group_id == uuid.UUID(performer_id)
                )
            )
            return [row[0] for row in result.all()]
        case _:
            return []
```

### Pattern 4: Work Item State Machine
**What:** Explicit state transitions for work items, extending the existing engine pattern.
**Valid transitions:**
```
AVAILABLE -> ACQUIRED   (acquire/claim)
ACQUIRED -> AVAILABLE   (release)
ACQUIRED -> COMPLETE    (complete/forward)
ACQUIRED -> REJECTED    (reject)
```
**Enforce with a transitions set**, same pattern as WORKFLOW_TRANSITIONS and ACTIVITY_TRANSITIONS in engine_service.

### Anti-Patterns to Avoid
- **Reimplementing advancement logic:** inbox_service must NOT contain workflow advancement code. Always delegate to engine_service.complete_work_item.
- **Lazy-loading in async context:** Per Phase 4 decision, always use selectinload. Never access unloaded relationships in async code.
- **Filtering in Python:** Apply all filters (state, priority, template_name) in the SQL query, not by filtering loaded results in Python.
- **Modifying engine_service.complete_work_item signature:** The existing function works. Add acquire-state checking in the inbox service layer, not by modifying the engine.

## Common Pitfalls

### Pitfall 1: PostgreSQL Enum Extension in Migrations
**What goes wrong:** Adding REJECTED to WorkItemState enum requires ALTER TYPE ... ADD VALUE, which cannot run inside a transaction in PostgreSQL.
**Why it happens:** PostgreSQL's ADD VALUE for enums is non-transactional.
**How to avoid:** In the Alembic migration, use `op.execute("ALTER TYPE workitemstate ADD VALUE IF NOT EXISTS 'rejected'")`. For SQLite tests, the enum is just a VARCHAR check constraint, so it works automatically.
**Warning signs:** Migration fails with "ALTER TYPE ... ADD VALUE cannot run inside a transaction block."

### Pitfall 2: Viewonly Relationship Limitations
**What goes wrong:** WorkItem.workflow_instance is a viewonly relationship (through activity_instances secondary table). Cannot use it for writes or in certain filter contexts.
**Why it happens:** The viewonly=True flag means SQLAlchemy won't track changes through it.
**How to avoid:** For inbox queries, join explicitly through ActivityInstance to reach WorkflowInstance. Don't rely on the viewonly relationship for filtering -- build the join in the query.
**Warning signs:** SAWarning about viewonly relationships or unexpected None values.

### Pitfall 3: Group Performer Creates Multiple Work Items
**What goes wrong:** If not handled carefully, group performer resolution could create duplicate work items or miss members.
**Why it happens:** D-06 requires one work item per group member. The current engine creates one work item per manual activity. Must change to loop over resolved performers.
**How to avoid:** resolve_performers returns a list. The engine loop creates one WorkItem per resolved user. If list is empty, create one with performer_id=None (unassigned).
**Warning signs:** Tests with group performers show wrong number of work items.

### Pitfall 4: Acquire Race Condition
**What goes wrong:** Two users simultaneously try to acquire the same AVAILABLE work item.
**Why it happens:** Without proper locking, both read state=AVAILABLE and both set state=ACQUIRED.
**How to avoid:** Use SELECT ... FOR UPDATE (SQLAlchemy with_for_update()) when loading the work item for acquire. In SQLite tests this is a no-op but is critical for PostgreSQL production. Alternatively, rely on the atomic UPDATE ... WHERE state='available' pattern.
**Warning signs:** Two users both successfully acquire the same item.

### Pitfall 5: Inbox Query Performance with Deep Joins
**What goes wrong:** The nested response (D-02) requires joining through 4-5 tables. Without proper indexing and eager loading, this becomes N+1 queries.
**Why it happens:** Each work item needs activity_instance -> activity_template + workflow_instance -> process_template + workflow_packages.
**How to avoid:** Use selectinload chains (not joinedload which creates cartesian products with collections). The query pattern shown in Architecture Pattern 2 handles this correctly.
**Warning signs:** Slow inbox queries, high query count in logs.

### Pitfall 6: complete_work_item Requires Workflow ID
**What goes wrong:** The existing engine_service.complete_work_item requires both workflow_id and work_item_id. The inbox UI only knows work_item_id.
**Why it happens:** The function was designed for the workflow-centric /workflows/{id}/work-items/{id}/complete endpoint.
**How to avoid:** In inbox_service, load the work item first to resolve the workflow_instance_id (through activity_instance), then pass both IDs to engine_service.complete_work_item. This is the "extend vs wrap" discretion item -- wrapping is the right approach.
**Warning signs:** 400 errors when trying to complete from inbox without workflow ID.

## Open Questions

1. **Should complete_work_item enforce ACQUIRED state?**
   - What we know: D-08 says "must acquire before completing." The existing engine_service.complete_work_item does not check for ACQUIRED state.
   - What's unclear: Should the enforcement be in inbox_service (wrapper) or engine_service (core)?
   - Recommendation: Add enforcement in inbox_service wrapper. Leave engine_service.complete_work_item unchanged so existing /workflows/{id}/work-items/{id}/complete endpoint still works without acquire (backward compatible). The inbox is stricter.

2. **How to construct the nested inbox response efficiently?**
   - What we know: D-02 requires activity template info + workflow summary + document list in one response.
   - What's unclear: Whether to use from_attributes with computed properties or manual construction in the service layer.
   - Recommendation: Build it in the service layer. Load work items with selectinload chains, then manually construct InboxItemResponse objects mapping nested data. Pydantic from_attributes won't naturally traverse the 4-table join path.
