  <action>
    Create `tests/test_lifecycle.py` with the following test functions. Use the existing test patterns from conftest.py (async_client, admin auth headers, etc.). Each test should create a document via upload first, then operate on it.

    **LIFE-01: Valid lifecycle transitions**

    `test_transition_draft_to_review`: Upload document (starts as DRAFT), POST transition to REVIEW, assert 200 and lifecycle_state="review" in response.

    `test_transition_review_to_approved`: Upload, transition to REVIEW, then transition to APPROVED. Assert 200.

    `test_transition_approved_to_archived`: Full chain: DRAFT -> REVIEW -> APPROVED -> ARCHIVED. Assert 200.

    `test_transition_review_to_draft_reject`: Upload, transition to REVIEW, then back to DRAFT. Assert 200 (reject back is a valid transition per D-02).

    `test_invalid_transition_draft_to_approved`: Upload, try to transition directly from DRAFT to APPROVED. Assert 400 (skipping REVIEW is invalid).

    `test_invalid_transition_archived_to_draft`: Full chain to ARCHIVED, then try back to DRAFT. Assert 400.

    `test_invalid_transition_draft_to_archived`: Try DRAFT -> ARCHIVED directly. Assert 400.

    **LIFE-02: Workflow-triggered lifecycle transition**

    `test_workflow_triggered_transition`: 
    - Create a process template with: START -> MANUAL (lifecycle_action="transition_to:review") -> END
    - Set lifecycle_action on the manual activity template (via API or direct DB setup)
    - Upload a document
    - Start workflow instance with document in package
    - Complete the manual activity's work item
    - Assert document lifecycle_state is now "review"

    `test_workflow_lifecycle_action_affects_all_package_documents`:
    - Same template with lifecycle_action
    - Upload TWO documents, attach both to workflow package
    - Complete activity
    - Assert BOTH documents transitioned (per D-10)

    `test_workflow_lifecycle_action_failure_does_not_halt`:
    - Create template with lifecycle_action="transition_to:approved" (invalid from DRAFT)
    - Upload document (DRAFT state)
    - Start workflow, complete activity
    - Assert workflow continues to next activity (not stuck), document still DRAFT (per D-04)

    **LIFE-03: Audit trail for lifecycle changes**

    `test_lifecycle_transition_creates_audit_record`:
    - Upload document, transition DRAFT -> REVIEW
    - Query audit records (GET /api/v1/audit or direct DB query)
    - Assert audit record exists with entity_type="document", action="lifecycle_transition", before_state containing "draft", after_state containing "review"

    `test_failed_transition_creates_audit_record`:
    - Upload document, attempt invalid transition
    - Assert audit record with action="lifecycle_transition_failed"

    **LIFE-04: ACL changes on lifecycle transition**

    `test_acl_changes_on_approval`:
    - Seed lifecycle_acl_rules for REVIEW -> APPROVED transition (remove WRITE for non_admin)
    - Upload document, add WRITE ACL for a second user
    - Transition DRAFT -> REVIEW -> APPROVED
    - Assert the WRITE ACL entry for the second user was removed
    - Assert ADMIN ACL for creator still exists

    Use existing conftest.py patterns for auth (login, get token, use headers). Follow the same httpx AsyncClient pattern used in test_documents.py.

    For workflow-triggered tests, follow the pattern from test_workflows.py: create template, add activities and flows, install template, start workflow, find work item, complete it.

  <action>
    Create `tests/test_acl.py` with the following test functions.

    **ACL-01: Documents have ACLs**

    `test_create_acl_entry`: Upload document (creates ADMIN ACL for creator). POST /documents/{id}/acl to add READ for another user. Assert 201.

    `test_list_acl_entries`: Upload document, list ACL entries via GET /documents/{id}/acl. Assert at least 1 entry (the ADMIN for creator).

    `test_delete_acl_entry`: Upload, list ACLs, delete one via DELETE /documents/{id}/acl/{acl_id}. Assert 200. List again, assert count decreased.

    `test_acl_permission_hierarchy`: Create ADMIN ACL for a user. Assert that user can access routes requiring READ, WRITE, DELETE (ADMIN implies all per D-06).

    `test_no_acl_entries_means_open_access`: 
    - This test needs a document with NO ACL entries (bypass the upload auto-creation or delete all entries)
    - Assert that any authenticated user can still access the document (backward compat)

    **ACL-02: Workflow activities modify ACLs via lifecycle**

    `test_workflow_acl_modification`:
    - Create template with activity having lifecycle_action="transition_to:approved"
    - Seed ACL rules for REVIEW -> APPROVED (remove WRITE for non_admin)
    - Upload document, add WRITE ACL for user2
    - Transition to REVIEW first (manually)
    - Start workflow, complete activity (triggers REVIEW -> APPROVED)
    - Assert user2's WRITE ACL was removed

    **ACL-03: ACL changes in audit trail**

    `test_acl_grant_creates_audit_record`: 
    - Upload document, add ACL entry for user2
    - Query audit records, assert entity_type="document_acl", action="acl_granted"

    `test_acl_removal_creates_audit_record`:
    - Upload document, add ACL, remove ACL
    - Assert audit record with action matching removal

    **ACL-04: Permission enforcement on API operations**

    `test_get_document_requires_read_permission`:
    - Create two users. Upload document as user1 (gets ADMIN ACL).
    - As user2 (no ACL entries for this document, BUT if no entries exist for the doc at all it is open, so we need at least one entry to trigger enforcement): the creator's ADMIN entry exists, so enforcement is active.
    - GET /documents/{id} as user2 should return 403 (user2 has no ACL entry, but entries exist on the doc).

    `test_update_document_requires_write_permission`:
    - Upload as user1. Add READ ACL for user2.
    - PUT /documents/{id} as user2 should return 403 (READ insufficient for WRITE).

    `test_checkout_requires_write_permission`:
    - Upload as user1. Add READ ACL for user2.
    - POST /documents/{id}/checkout as user2 should return 403.

    `test_upload_does_not_require_permission`:
    - Any authenticated user can POST /documents/ (no document_id to check against).
    - Assert 201.

    `test_list_documents_no_permission_check`:
    - GET /documents/ as any authenticated user should return 200 (no per-document ACL on list).

    `test_admin_permission_grants_all_access`:
    - Upload as user1 (gets ADMIN). As user1: GET, PUT, checkout, checkin all succeed.

    `test_group_based_permission`:
    - Create a group, add user2 to group.
    - Upload document as user1. Add READ ACL with principal_type="group" and principal_id=group_id.
    - GET /documents/{id} as user2 should return 200 (group-based permission).

    For creating a second user, follow conftest.py patterns. The test will need to:
    1. Register/create user2 via API
    2. Login as user2 to get auth token
    3. Use user2's token for requests that should be denied/allowed
