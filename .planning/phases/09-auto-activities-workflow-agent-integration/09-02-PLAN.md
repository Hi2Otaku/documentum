<objective>
Create the Celery app module, Workflow Agent tasks (poll + execute), and admin retry/skip endpoints.

Purpose: This plan wires the auto method infrastructure from Plan 01 into the Celery task queue. The poll task (Celery beat, every 10 seconds per D-04) finds ACTIVE AUTO activities and dispatches individual execution tasks. Each execution task runs the registered method, logs the result, handles errors with retry/backoff (D-05), and advances the workflow on success. Admin endpoints (D-07) allow retry or skip of failed activities.

Output: Complete Workflow Agent capable of executing auto activities with error handling.
</objective>

    **`src/app/tasks/auto_activity.py`**:

    Two Celery tasks:

    1. `poll_auto_activities` — periodic task (beat schedule every 10s per D-04):
       - Uses `asyncio.run(_poll_async())` to bridge to async
       - `_poll_async()`:
         - Creates session via `async_session_factory()`
         - Queries `ActivityInstance` JOIN `ActivityTemplate` WHERE `ActivityInstance.state == ActivityState.ACTIVE` AND `ActivityTemplate.activity_type == ActivityType.AUTO`
         - For locking: check if dialect is sqlite; if not, add `.with_for_update(skip_locked=True)` to prevent duplicate dispatch. If sqlite, skip locking (tests use sqlite).
         - For each found activity, dispatch `execute_auto_activity.delay(str(ai.id), str(ai.workflow_instance_id))`
         - Commit and close session

    2. `execute_auto_activity(self, activity_instance_id: str, workflow_instance_id: str)` — bound task with `bind=True`:
       - Uses `asyncio.run(_execute_async(activity_instance_id, workflow_instance_id))` 
       - `_execute_async(activity_instance_id, workflow_instance_id)`:
         - Creates session via `async_session_factory()`
         - Load ActivityInstance (with selectinload of activity_template)
         - Guard: if state is not ACTIVE, return (already processed or cancelled)
         - Load ActivityTemplate via the relationship
         - Guard: if `activity_type != AUTO`, return
         - Get method via `get_auto_method(activity_template.method_name)`
         - If method is None, raise `ValueError(f"Auto method '{activity_template.method_name}' not registered")`
         - Load WorkflowInstance
         - Load ProcessVariables for this workflow (query by workflow_instance_id, is_deleted=False)
         - Build variable dict using `_resolve_variable_value` pattern (import from engine_service or duplicate the simple match logic)
         - Load document_ids from WorkflowPackage (query by workflow_instance_id)
         - Create `ActivityContext` with all loaded data, `user_id="system"`
         - Create `AutoActivityLog` entry with status="running", started_at=now, attempt_number (track via log count + 1)
         - Execute: `result = await method(ctx)`
         - On success:
           - Update log: status="success", completed_at=now, result_data=result
           - Load full template with selectinload (activity_templates, flow_templates, process_variables)
           - Build template_to_instance mapping (query all ActivityInstances for this workflow)
           - Reload process variables
           - Call `_advance_from_activity(db, workflow, activity_instance, template, template_to_instance, "system", instance_variables=variables)` to advance the workflow
           - Commit
         - On exception:
           - Update log: status="error", error_message=str(e), error_traceback=traceback.format_exc(), completed_at=now
           - Increment attempt count check: query count of AutoActivityLog entries for this activity_instance_id
           - If attempts < 3 (per D-05): rollback, then `raise self.retry(exc=e, countdown=backoff)` where backoff = 10 * (3 ** (attempt - 1)) giving 10s, 30s, 90s
           - If attempts >= 3: mark activity_instance.state = ERROR, commit, log error
           - On `MaxRetriesExceededError` (from Celery): mark activity_instance.state = ERROR, commit
         - Timeout handling: use `soft_time_limit=60` as default. Catch `SoftTimeLimitExceeded` from celery.exceptions, treat as error with status="timeout"

    Import `_advance_from_activity` from `app.services.engine_service` using lazy import inside the async function to avoid circular imports.

    Important: the `_advance_from_activity` function is private (underscore prefix). Import it directly since Celery tasks are internal infrastructure, not external API consumers. Alternatively, if the executor prefers, create a thin public wrapper `advance_auto_activity` in engine_service that calls `_advance_from_activity` with the same signature. Either approach is acceptable.

    **`src/app/routers/workflows.py`** (per D-07):
    Add two new endpoints:

    1. `POST /api/v1/workflows/{workflow_id}/activities/{activity_id}/retry`:
       - Depends on `get_db`, `get_current_user`
       - Load ActivityInstance by `activity_id`, verify it belongs to `workflow_id`
       - Verify state is `ActivityState.ERROR`; if not, raise HTTPException 400 "Activity is not in error state"
       - Reset state to `ActivityState.ACTIVE` (this transition exists in ACTIVITY_TRANSITIONS)
       - Create audit record: action="auto_activity_retried", entity_type="activity_instance"
       - Return `EnvelopeResponse(data=ActivityRetryResponse(activity_instance_id=..., status="requeued", message="Activity requeued for execution"))`
       - The poll task will pick it up on next cycle

    2. `POST /api/v1/workflows/{workflow_id}/activities/{activity_id}/skip`:
       - Depends on `get_db`, `get_current_user`
       - Load ActivityInstance by `activity_id`, verify it belongs to `workflow_id`
       - Verify state is `ActivityState.ERROR`; if not, raise HTTPException 400 "Activity is not in error state"
       - Load full template with relations (same pattern as complete_work_item)
       - Build template_to_instance mapping
       - Load process variables
       - Mark activity COMPLETE (direct state set, since ERROR->ACTIVE->COMPLETE would need two steps; add (ERROR, COMPLETE) to ACTIVITY_TRANSITIONS in engine_service if needed, or transition ERROR->ACTIVE then ACTIVE->COMPLETE)
       - Call `_advance_from_activity` to advance workflow past the skipped activity
       - Create `AutoActivityLog` entry with status="skipped"
       - Create audit record: action="auto_activity_skipped", entity_type="activity_instance"
       - Return `EnvelopeResponse(data=ActivityRetryResponse(activity_instance_id=..., status="skipped", message="Activity skipped, workflow advanced"))`
