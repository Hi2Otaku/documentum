**Part A — Add Redis publish to notification_service.create_notification():**

This is the most robust placement because it covers ALL notification creation paths (not just event handlers).

In `src/app/services/notification_service.py`:
1. Add imports at top: `import json` and `import redis.asyncio as aioredis`
2. Add a module-level `_redis_client = None` and an async helper `_get_redis()` that lazily initializes from `settings.redis_url` (same pattern used in event_bus.py)
3. After the existing `await db.flush()` line in `create_notification()`, add Redis publish:

```python
# Publish to Redis for SSE real-time delivery
try:
    r = await _get_redis()
    payload = json.dumps({
        "id": str(notification.id),
        "user_id": str(user_id),
        "title": title,
        "message": message or "",
        "notification_type": notification_type,
        "entity_type": entity_type,
        "entity_id": str(entity_id) if entity_id else None,
        "created_at": notification.created_at.isoformat() if notification.created_at else None,
    })
    await r.publish("notifications", payload)
except Exception:
    logger.warning("Failed to publish notification to Redis, SSE delivery skipped", exc_info=True)
```

The try/except ensures Redis failure does not break notification creation. The SSE generator in notifications.py filters by `data.get("user_id") == str(user_id)` so the `user_id` field is required.
