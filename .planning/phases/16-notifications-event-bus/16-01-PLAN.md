# Phase 16 Plan 01: Event Bus & In-App Notifications

## Objective

Build the domain event bus with persistent storage and the in-app notification system including backend API and frontend notification bell UI. This establishes the event infrastructure that all subsequent v1.2 phases depend on.

## Context

- @src/app/models/ -- existing SQLAlchemy model patterns
- @src/app/routers/ -- existing FastAPI router patterns
- @src/app/services/ -- existing service layer patterns
- @frontend/src/components/layout/ -- AppShell where notification bell goes

## Tasks

### Task 1: Domain Event model and event bus service
type="auto"

Create the DomainEvent SQLAlchemy model and EventBus service that can emit and persist events.

**Done criteria:**
- DomainEvent model with event_type, payload (JSONB), actor_id, entity_type, entity_id, created_at
- Alembic migration for domain_events table
- EventBus service with emit() method that persists events
- Model registered in __init__.py

### Task 2: Notification model and service
type="auto"

Create the Notification model and NotificationService for creating and managing in-app notifications.

**Done criteria:**
- Notification model with user_id, title, message, notification_type, is_read, entity_type, entity_id
- Alembic migration for notifications table
- NotificationService with create, list (with pagination), mark_read, mark_all_read, get_unread_count
- Event handlers that create notifications on work_item.assigned and work_item.delegated events

### Task 3: Notifications API router
type="auto"

Create the FastAPI router for notification endpoints.

**Done criteria:**
- GET /notifications -- list user notifications (paginated, filterable by is_read)
- GET /notifications/unread-count -- returns unread count
- PUT /notifications/{id}/read -- mark single as read
- PUT /notifications/mark-all-read -- mark all as read
- All endpoints require authentication

### Task 4: Integrate event emission into existing services
type="auto"

Wire event emission into existing workflow and document services so domain events fire on key state changes.

**Done criteria:**
- workflow started/completed/failed emits domain events
- work item assigned/completed/delegated emits domain events
- document uploaded emits domain event
- lifecycle state changed emits domain event
- Events are persisted in domain_events table

### Task 5: Notification bell UI component
type="auto"

Build the notification bell icon with unread badge in the app header, and a dropdown list of notifications.

**Done criteria:**
- NotificationBell component with bell icon and unread count badge
- Dropdown/popover showing recent notifications
- Mark as read on click, mark all read button
- Integrated into AppShell/Sidebar header
- API hooks using TanStack Query

## Verification

- Domain events table exists and events are persisted on service calls
- Notification endpoints return correct data with auth
- Frontend notification bell renders with unread count

## Success Criteria

- EVENT-01: System emits domain events for document uploads, lifecycle changes, workflow state transitions
- EVENT-02: Events are persisted in a durable event table
- NOTIF-01: User receives in-app notification when work item assigned
- NOTIF-02: User receives in-app notification when task delegated
- NOTIF-05: User can view notification list with unread count badge
- NOTIF-06: User can mark notifications as read individually or in bulk
