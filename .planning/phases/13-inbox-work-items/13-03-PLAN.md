**Create `frontend/src/components/inbox/CompleteDialog.tsx`** (per D-05):

Props: `{ workItemId: string; open: boolean; onOpenChange: (open: boolean) => void }`

- Dialog with title "Complete Task"
- Optional Textarea with placeholder "Add a comment (optional)"
- Cancel button (variant="outline") and "Complete" button (variant="default")
- On confirm:
  1. If comment text is non-empty (trimmed), call `addComment(workItemId, comment)` first
  2. Then call `completeWorkItem(workItemId, { output_variables: {}, selected_path: null, next_performer_id: null })`
  3. Use `useMutation` from TanStack Query for the complete call
  4. On success: close dialog, invalidate `["inbox"]` queries via `useQueryClient().invalidateQueries({ queryKey: ["inbox"] })`
  5. On error: keep dialog open, log error (console.error)
- "Complete" button shows spinner/disabled while mutation is pending

  1. Call `useAuthStore.getState().setAvailability(false)` — but since setAvailability already calls the API, instead directly update the store: `useAuthStore.setState({ isAvailable: false })`

**Create `frontend/src/components/inbox/QueueItemsTable.tsx`** (per D-12):

Props: `{ queueId: string }`

- Fetch queue detail: `useQuery({ queryKey: ["queues", queueId], queryFn: () => fetchQueueDetail(queueId), enabled: !!queueId })`

Note: The queue detail endpoint returns members, not work items. The backend doesn't have a "queue work items" endpoint. For now, show the queue members list with their info. The "Claim" concept works through the inbox — items assigned to a queue appear as available work items that any queue member can acquire.

Actually, re-reading the CONTEXT.md D-11/D-12: "Click a queue to see its unclaimed items" and "Claiming moves the item to the user's personal inbox." The claim action uses `POST /api/inbox/{id}/acquire`. But there's no endpoint to list work items FOR a specific queue. The inbox list endpoint filters by current user, not by queue.

**Pragmatic approach for queue items:** Since there's no backend endpoint for "items in queue X", render the queue detail (name, description, member list) and show a message: "Queue work items appear in your My Inbox tab when you are a member of this queue. Switch to My Inbox to view and acquire available items."

This is honest about the current backend limitation while still providing useful queue browsing (D-11). The Claim button concept merges with the Acquire button in My Inbox.

**Queue items panel layout:**
- When no queue selected: centered text "Select a queue" (heading) + "Click a queue from the list to view its details." (body muted)

  - Header: queue name (text-base font-semibold) + description (text-sm text-muted-foreground)
  - Separator
  - "Members" heading (text-sm font-semibold) + list of members (username, email)
  - Each member: flex row, text-sm, with Avatar showing initials
  - Info message at bottom: "Queue work items appear as available items in your My Inbox tab." (text-xs text-muted-foreground, italic)

**Empty queue items:** `<InboxEmptyState heading="No unclaimed items" body="All items in this queue have been claimed. Check back later." />`
