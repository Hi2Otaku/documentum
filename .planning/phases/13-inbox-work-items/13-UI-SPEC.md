## Interaction Contracts

### Table Row Selection (D-01)

1. User clicks a table row -> row becomes selected (highlighted background + left border).
2. Detail panel loads that work item's details via `GET /api/inbox/{id}`.
3. Only one row selected at a time.
4. Selected row state stored in component state (not URL, not Zustand).
5. On page change (pagination), selection clears and detail panel shows "Select a work item".

### State Filter (D-03)

1. Select dropdown above table with options: "All", "Available", "Acquired", "Delegated".
2. Default: "All" (shows available + acquired + delegated -- excludes complete/rejected).
3. Changing filter resets pagination to page 1 and clears selection.
4. Filter value passed as `?state=` query param to `GET /api/inbox`.

### Acquire (D-07)

1. Detail panel shows "Acquire" button when work item state is `available`.
2. Click button -> immediate `POST /api/inbox/{id}/acquire` (no dialog).
3. On success: optimistic update -- badge changes to "acquired", Acquire button hides, Complete/Reject/Delegate buttons appear.
4. On error: toast notification with error message from API.

### Complete (D-05)

1. Detail panel shows "Complete" button when work item state is `acquired` and performer is current user.
2. Click opens CompleteDialog.
3. Dialog contents: "Complete Task" heading, optional comment Textarea (placeholder: "Add a comment..."), Cancel + "Complete" buttons.
4. "Complete" button calls `POST /api/inbox/{id}/complete` with `{ output_variables: {}, selected_path: null, next_performer_id: null }`.
5. If comment was entered, also call `POST /api/inbox/{id}/comments` with content (before complete, so comment is recorded).
6. On success: remove item from table (or update state to "complete"), clear selection, show success toast.
7. On error: keep dialog open, show error in toast.

### Reject (D-06)

1. Detail panel shows "Reject" button when work item state is `acquired` and performer is current user.
2. Click opens RejectDialog.
3. Dialog contents: "Reject Task" heading, required comment Textarea (placeholder: "Reason for rejection (required)"), Cancel + "Reject" buttons.
4. "Reject" button is disabled until Textarea has content (min 1 char, trimmed).
5. Calls `POST /api/inbox/{id}/reject` with `{ reason: commentText }`.
6. On success: remove item from table (or update state to "rejected"), clear selection, show success toast.
7. On error: keep dialog open, show error in toast.

### Delegate (D-09)

1. Detail panel shows "Delegate" button when work item state is `acquired` and performer is current user.
2. Click opens DelegateDialog.
3. Dialog contents: "Delegate Tasks" heading, explanatory text "Setting a delegate will route all your future tasks to the selected user. You will be marked as unavailable.", user picker Select (fetches from `GET /api/users`), Cancel + "Confirm Delegation" buttons.
4. "Confirm Delegation" disabled until a user is selected.
5. Calls `PATCH /api/users/{currentUserId}/availability` with `{ is_available: false, delegate_id: selectedUserId }`.
6. On success: update authStore `isAvailable` to false, show success toast, refresh inbox (items may have been re-routed).
7. On error: keep dialog open, show error in toast.

Note: The backend supports user-level delegation only (not per-work-item). The UI honestly communicates this -- delegating affects all future tasks, not just the selected one. The "Delegate" button is placed in the detail panel per D-09 but the dialog clarifies the scope.

### Claim from Queue (D-12)

1. In Queues tab, user clicks a queue to see its unclaimed items.
2. Each item row has a "Claim" button (small, outline variant).
3. Click "Claim" -> `POST /api/inbox/{id}/acquire`.
4. On success: item disappears from queue list, appears in user's My Inbox tab. Show toast "Task claimed".
5. On error: toast with error message.

### Pagination

1. Page size: 20 items (matches API default).
2. Pagination controls at table footer: "Page X of Y", Previous/Next buttons.
3. Previous disabled on page 1. Next disabled on last page.
4. Page change triggers `GET /api/inbox?skip={page*20}&limit=20&state={filter}`.

### Loading States

1. Table loading: 5 Skeleton rows, each 48px tall, spanning all columns.
2. Detail panel loading: Skeleton blocks matching header (1 line), context card (3 lines), actions (button-sized blocks).
3. Initial page load: both table and detail panel show skeletons simultaneously.

## State Management

### Server State (TanStack Query)

| Query Key | Endpoint | Refetch |
|-----------|----------|---------|
| `["inbox", { state, skip, limit }]` | `GET /api/inbox` | On filter change, on page change, after mutation |
| `["inbox", workItemId]` | `GET /api/inbox/{id}` | On row selection |
| `["inbox", workItemId, "comments"]` | `GET /api/inbox/{id}/comments` | After add comment mutation |
| `["queues"]` | `GET /api/queues` | On Queues tab activation |
| `["users"]` | `GET /api/users` | On DelegateDialog open (for user picker) |

### Mutations

| Mutation | Endpoint | Invalidates |
|----------|----------|-------------|
| acquireWorkItem | `POST /api/inbox/{id}/acquire` | `["inbox"]`, `["inbox", id]` |
| completeWorkItem | `POST /api/inbox/{id}/complete` | `["inbox"]` |
| rejectWorkItem | `POST /api/inbox/{id}/reject` | `["inbox"]` |
| addComment | `POST /api/inbox/{id}/comments` | `["inbox", id, "comments"]` |
| setDelegation | `PATCH /api/users/{id}/availability` | `["inbox"]` |

### Client State (Component-level)

| State | Type | Scope |
|-------|------|-------|
| selectedWorkItemId | `string | null` | InboxPage component |
| activeTab | `"inbox" | "queues"` | InboxPage component |
| selectedQueueId | `string | null` | InboxPage component (Queues tab) |
| stateFilter | `string` | InboxPage component |
| currentPage | `number` | InboxPage component |

No new Zustand stores needed. All inbox state is local to the page component tree.
