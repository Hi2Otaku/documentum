# Phase 23 Plan 01: Digital Signatures - PKCS7/CMS Signing, Verification, and Immutability

## Objective

Implement cryptographic digital signatures on document versions using PKCS7/CMS format. Users can sign document versions, verify signatures, list all signatures on a document, and the system enforces immutability on signed versions (no re-upload, check-in, or metadata modification).

## Context

- @src/app/models/document.py - Document and DocumentVersion models
- @src/app/services/document_service.py - Document CRUD and versioning logic
- @src/app/routers/documents.py - Document API endpoints
- @src/app/schemas/document.py - Pydantic schemas
- @src/app/models/__init__.py - Model registry
- @src/app/main.py - Router registration

## Tasks

### Task 1: Add cryptography dependency and DigitalSignature model
type="auto"

**Do:**
1. Add `cryptography` to pyproject.toml dependencies
2. Create `src/app/models/signature.py` with `DigitalSignature` model:
   - id (UUID, PK)
   - document_version_id (FK to document_versions.id)
   - signer_id (FK to users.id)
   - signature_data (LargeBinary - the PKCS7/CMS signature bytes)
   - certificate_pem (Text - the signing certificate PEM)
   - digest_algorithm (String - e.g., "sha256")
   - signed_at (DateTime with timezone)
   - is_valid (Boolean - last verification result)
   - Common BaseModel fields
3. Add `is_signed` Boolean column to DocumentVersion model (default False)
4. Register model in `__init__.py`
5. Create Alembic migration

**Done when:** Migration runs and DigitalSignature table exists.

### Task 2: Signature service - sign and verify
type="auto"

**Do:**
1. Create `src/app/services/signature_service.py` with:
   - `sign_document_version(db, document_id, version_id, user_id, private_key_pem, certificate_pem)` - creates PKCS7/CMS signature of the version content, stores DigitalSignature record, marks version as signed
   - `verify_signature(db, signature_id)` - re-downloads content, verifies PKCS7 signature against stored certificate, updates is_valid
   - `list_signatures(db, document_id, version_id)` - returns all signatures for a version
   - `get_signature(db, signature_id)` - get single signature
2. Use `cryptography` library for PKCS7 signing (CMS/PKCS7 with SHA-256)
3. Download version content from MinIO for signing/verification

**Done when:** Service functions create valid PKCS7 signatures and verify them.

### Task 3: Signature schemas and API endpoints
type="auto"

**Do:**
1. Create `src/app/schemas/signature.py` with:
   - `SignRequest` (private_key_pem, certificate_pem)
   - `SignatureResponse` (id, document_version_id, signer_id, digest_algorithm, signed_at, is_valid, certificate_subject, created_at)
   - `VerifyResponse` (signature_id, is_valid, details)
2. Create `src/app/routers/signatures.py` with:
   - `POST /documents/{document_id}/versions/{version_id}/sign` - sign a version
   - `GET /documents/{document_id}/versions/{version_id}/signatures` - list signatures
   - `POST /signatures/{signature_id}/verify` - verify a specific signature
3. Register router in main.py

**Done when:** API endpoints for sign, verify, and list work.

### Task 4: Enforce immutability on signed versions
type="auto"

**Do:**
1. In `document_service.checkin_document` - before creating new version, check if document has any signed versions and block re-upload to signed versions
2. In `document_service.update_document_metadata` - block metadata changes if document has signed current version
3. Add guard in checkout that warns/blocks if current version is signed
4. Return clear HTTP 409 error messages explaining signed immutability
5. Add audit records for blocked operations

**Done when:** Attempting to check-in, re-upload, or modify metadata on a signed document version returns a clear error.

### Task 5: Add is_signed field to version response and update version schema
type="auto"

**Do:**
1. Add `is_signed` to `DocumentVersionResponse` schema
2. Update version list endpoint to show signature status
3. Add signature count to version response if possible

**Done when:** Version responses include is_signed field.

## Verification

- Signing a document version produces a valid PKCS7/CMS signature
- Verifying a valid signature returns is_valid=true
- Listing signatures shows signer, timestamp, and validity
- Attempting to check-in a new version over a signed version returns 409
- Attempting to modify metadata of a document with signed current version returns 409

## Success Criteria

1. User can digitally sign a specific document version using PKCS7/CMS
2. User can verify the validity of any signature on a signed document version
3. User can view a list of all signatures on a document with signer identity, timestamp, and validity status
4. System prevents re-upload, check-in, or metadata modification of a signed document version
